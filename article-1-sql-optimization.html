How I Cut SQL Query Time from 45 Seconds to 8 Seconds

Daniel Nnadi · 4 min read · SQL Performance Optimization

When I joined the data team at an automotive parts company, the inventory query everyone relied on took 45 seconds to run. Analysts had built workarounds pulling data in the morning and working from yesterday's snapshot. Real-time decision-making was impossible.

The Problem

The database was SQL Server with inventory, suppliers, purchase_orders, and stock_movements tables. The query joining all four was doing full table scans on every execution.

What I Found

  • No composite indexes filters on warehouse_id and category_code had no indexes
  • Implicit type conversions VARCHAR compared to INT, adding about 8 seconds
  • Correlated subquery running 500+ times per query
  • No covering indexes excessive key lookups
"The most expensive queries are often not the complex ones they are the simple ones running against a schema never designed for how the data is actually queried."

The Fix: Index Redesign

CREATE NONCLUSTERED INDEX IX_inventory_warehouse_category
ON inventory (warehouse_id, category_code)
INCLUDE (sku_id, product_name, qty_on_hand, reorder_threshold);

CREATE NONCLUSTERED INDEX IX_po_sku_delivery
ON purchase_orders (sku_id)
INCLUDE (delivery_date);

Key Lessons

  • Read the execution plan before writing a single fix
  • Correlated subqueries in SELECT are almost always wrong use CTEs instead
  • Design indexes for your queries, not your tables
